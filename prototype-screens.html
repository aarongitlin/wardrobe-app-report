Prototype Scope

The prototype simulates the full first-week experience: onboarding, seeing your closet populate automatically, browsing items, building an outfit, and checking your activity feed. All data is fake but realistic ‚Äî 30+ clothing items across categories, pre-built outfits, and a simulated activity history showing auto-added items over time.

What's Real vs. Simulated

Real: All UI interactions (taps, filters, drag-and-drop), navigation between screens, visual design, layout, and information architecture.

Simulated: Email connection (fake OAuth), email scanning (animated progress), item data (pre-loaded dummy items), background removal (pre-cut images).

04 ‚Äî Auto-Detection
The Magic Moment
When the app finds a new purchase in your email and presents it for confirmation. This is the "wow" interaction ‚Äî the thing that makes this product different from everything else.

New Item Detected

This screen appears either from a push notification or when the user taps a new item in the Activity feed. It shows the extracted item image with background removed, alongside auto-detected metadata. The user can confirm with one tap or edit any field.

  • tap
    "Add to Closet" ‚Äî confirms the item with current tags. Item appears in the closet grid with a brief celebratory animation (subtle confetti or glow).
  • tap
    "Edit tags" ‚Äî fields become editable inline. Category becomes a dropdown, color shows a palette picker, brand and price are text fields.
  • tap
    Category/Color/Brand fields ‚Äî tap any field to quick-edit it directly. Most users will confirm without editing.
  • ‚Üê
    Swipe to dismiss ‚Äî dismisses the item (doesn't add to closet). "Are you sure?" confirmation appears.
In the prototype: This screen is triggered by tapping a "Simulate new item" button in the Profile tab (for demo purposes). It runs a brief animation of "detecting" a new item, then reveals the confirmation screen. The item data is pre-defined but presented as if just discovered.
05 ‚Äî Outfit Builder
Build & Browse Outfits
Two sub-screens: the saved outfits gallery and the drag-and-drop outfit builder. The builder is the most interaction-heavy screen in the app.

Saved Outfits Gallery

2-column grid of saved outfit collages. Each card shows a mini-preview of the outfit with name and item count. Tapping opens the outfit in the builder for editing. The "+ New" button opens a fresh canvas.

Outfit Builder

Drag items from the bottom tray onto the canvas. Resize with corner handles. Items snap to a loose grid for alignment. The active item shows selection handles (dashed accent border). Category tabs filter the item tray.

Builder Interactions

Drag to add: Long-press an item in the tray, drag it up onto the canvas. It snaps into place with a subtle shadow lift.

Resize: Tap an item on the canvas to select it (shows handles). Drag corners to resize proportionally.

Reposition: Drag a selected item to move it around the canvas.

Remove: Drag an item below the canvas or tap the X that appears when selected.

Save Flow

Tapping "Save" opens a sheet asking for an outfit name (text input with auto-suggested names like "Monday Look" or "Casual Weekend"). Optional: add a date to schedule it. The saved outfit appears in the gallery.

Prototype note: Drag-and-drop will be fully functional in the prototype. We'll use a simple physics model for smooth dragging. Pre-loaded items will use the same simplified clothing silhouettes from the closet grid.

  • ‚ö°

Emotional design: This screen is where the "it just works" feeling lives. Seeing items auto-populate over time (even with fake data in the prototype) creates the sense of effortless wardrobe management. The green dots are deliberately satisfying ‚Äî each one is a moment where the user did nothing and the app did the work.

08 ‚Äî Implementation Notes
Building the Prototype
Technical approach and what to prioritize for the interactive prototype.

Tech Stack

Single-page React app with the Warm Studio design system. All data is hardcoded JSON ‚Äî no backend needed. Routing simulated with React state (no URL changes). Animations via CSS transitions and Framer Motion for drag-and-drop.

Deployed to the same Vercel instance as the report, accessible via a separate URL path. Mobile-first but responsive ‚Äî should work in a phone browser for realistic testing.

Priority Order

Must have: Onboarding flow, closet grid with working filters, item detail view, tab navigation

Should have: Activity feed, outfit gallery, simulated auto-detection demo

Nice to have: Drag-and-drop outfit builder, search, scroll animations, haptic-style micro-interactions

Dummy Data Plan

~30 clothing items across categories: 12 tops (shirts, sweaters, tees), 8 bottoms (chinos, jeans, shorts), 6 shoes (sneakers, boots, dress shoes), 5 outerwear (jackets, blazers). Each item has: name, brand, price, color, category, date added, source (auto/manual), and a simplified silhouette image. 4 pre-built outfits with names and dates.

User Testing Goals

Key questions to answer with this prototype:

1. Does the email connect flow feel trustworthy?
2. Is the closet grid easy to browse and filter?
3. Does the auto-detection concept resonate emotionally?
4. Is the information architecture intuitive (can users find what they need)?
5. What's missing that users expect to see?
